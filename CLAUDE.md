# CLAUDE.md

Weaver is a GitHub Action for organization-led markdown templating. It runs
from a central repo, renders `*.md` templates, and injects them between
`<!-- weaver:<name>:START/END -->` markers in files across the org's other
repos — via the GitHub API only (no cloning) — opening one PR per repo.

## Architecture

Plain JavaScript, native ESM (`"type": "module"`), Node >= 24, no TypeScript —
types are expressed as JSDoc annotations. Yarn 4 (`yarn`, not npm).

- [src/index.js](src/index.js) — entry shell: builds the Octokit client, calls `run()`, reports failures via `core.setFailed`. Excluded from coverage.
- [src/main.js](src/main.js) — orchestrator: loads config + templates, resolves repos, renders, syncs each repo, sets outputs.
- [src/config.js](src/config.js) — resolves action inputs into one config object. There is no config file; every setting is an input (see [action.yml](action.yml)).
- [src/template.js](src/template.js) — minimal in-house renderer: `{{ key }}` (escaped), `{{{ key }}}` (raw), `{{#if key}}…{{/if}}`, dotted paths.
- [src/markers.js](src/markers.js) — finds paired marker comments and replaces content between them. Markers inside fenced code blocks are documentation and must be ignored.
- [src/repos.js](src/repos.js) — resolves target repos via the REST API; builds per-repo template variables (`repo.*`, `org.*`).
- [src/commit.js](src/commit.js) — pushes changes to a head branch and opens/reuses one PR per repo. Never commits to the base branch.
- [src/retry.js](src/retry.js) — retries transient GitHub failures (secondary rate limits, 5xx); genuine 4xx errors re-throw immediately.

`dist/` is generated by `ncc` at release time — never edit it by hand.

## Commands

```sh
yarn lint      # eslint (also lints json/markdown)
yarn format    # eslint --fix
yarn test      # jest (native ESM via --experimental-vm-modules)
yarn coverage  # jest with the 80% coverage threshold
yarn build     # ncc bundle to dist/index.mjs
```

## Testing

- Tests live in [test/](test/), one file per `src` module, named `<module>.test.js`.
- `@actions/core` and `@actions/github` are redirected to hand-written manual
  mocks in [**mocks**/](__mocks__/) via `moduleNameMapper` — do not use
  `jest.mock()` / `unstable_mockModule` for them.
- Coverage threshold is 80% (branches/functions/lines); `src/index.js` is
  excluded. New logic needs tests to keep the gate green.

## Commits and pull requests

Use Conventional Commits: `<type>(<optional-scope>): <imperative subject>`
(lowercase, ≤100 chars, no trailing period). Types: feat, fix, docs, chore,
refactor, test. Body paragraphs wrapped ~72–80 chars, written in plain prose:
what changed and why, then behavior/mechanics, then edge cases and tests.
Do NOT add a "Generated with Claude Code" / Co-Authored-By footer.

Releases use Changesets, not semantic-release. Any user-facing change needs a
changeset (`yarn changeset`); its text is lifted verbatim into CHANGELOG.md
and release notes, so write it for a human reader — lead with the outcome,
then mechanics, edge cases, and test coverage (see CHANGELOG.md for the
house style). PR titles follow the same conventional format; PR bodies
summarize behavior changes and call out any new inputs/outputs, which must
also be documented in both `action.yml` and the README tables.

## Style

- **DO:** keep code self-documenting; when a comment is needed, keep it brief
  and explain only the _why_ the code can't show — never restate what the
  code does.
- Public functions carry JSDoc with `@param`/`@returns` types (this is the
  project's only typing layer — keep it accurate).
- Tabs for indentation; Prettier + ESLint enforce formatting. Husky +
  lint-staged run both on commit — don't bypass with `--no-verify`.
- The README header/footer sections between `weaver:*` markers are
  auto-generated by Weaver itself — do not edit between those markers.
